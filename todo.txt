1. FILE / ARGUMENT HANDLING
------------------------------------------------------------------------------
[ ] parse_scene(t_game *game, char *path)   <-- main entry point
    [ ] Check extension is exactly ".cub" (reject ".cube", "cub", etc.)
    [ ] Open the file (open + O_RDONLY); error on failure
    [ ] Reject if path is a directory / cannot be read
    [ ] Read the file line by line with get_next_line
    [ ] Dispatch each line:
        - texture identifier (NO/SO/WE/EA)  -> parse_textures
        - color identifier   (F/C)          -> parse_colors
        - map content                       -> collect for parse_map
        - empty line                        -> allowed BEFORE map only
    [ ] Enforce: all 6 identifiers (NO SO WE EA F C) present before the map
    [ ] Enforce: map is the LAST block in the file (nothing after it)
    [ ] Call validate_map() at the end
    [ ] Free the gnl line every iteration (no leaks)
    [ ] Always close the fd

------------------------------------------------------------------------------
2. TEXTURE PARSING  (parse_textures)
------------------------------------------------------------------------------
[ ] Recognize identifiers: NO, SO, WE, EA
[ ] Store path into config.tex_path[NORTH/SOUTH/WEST/EAST] (use enum e_dir)
[ ] Trim leading/trailing whitespace around the path
[ ] Reject duplicate identifier (same one given twice)
[ ] Reject missing path after identifier
[ ] Reject garbage after the path (e.g. "NO a.xpm b.xpm")
[ ] Duplicate the string (ft_strdup) so it survives gnl line free
[ ] (Optional now) sanity check path ends in ".xpm"
[ ] Return error code cleanly on any failure

------------------------------------------------------------------------------
3. COLOR PARSING  (parse_colors)
------------------------------------------------------------------------------
[ ] Recognize identifiers: F (floor), C (ceiling)
[ ] Split the "R,G,B" values on ','
[ ] Require exactly 3 components
[ ] Each component: digits only, range 0-255
[ ] Trim whitespace around each number
[ ] Pack into int:  (R << 16) | (G << 8) | B
[ ] Store in config.floor_color / config.ceiling_color
[ ] Reject duplicate F or C
[ ] Reject out-of-range / non-numeric / missing components
[ ] Track "was this color set?" so missing F or C is caught

------------------------------------------------------------------------------
4. MAP PARSING  (parse_map)
------------------------------------------------------------------------------
[ ] Collect all map lines into config.map (char **)
[ ] Preserve spaces EXACTLY (do NOT trim map lines)
[ ] Do NOT strip the trailing '\n' silently -> handle it consistently
[ ] Reject empty lines INSIDE the map body (map must be contiguous)
[ ] Compute config.map_height (number of rows)
[ ] Compute config.map_width  (length of the LONGEST row)
[ ] (Recommended) pad shorter rows to map_width with spaces
    -> makes flood-fill / bounds checks much simpler
[ ] NULL-terminate the char ** array
[ ] Handle allocation failures without leaking

------------------------------------------------------------------------------
5. MAP VALIDATION  (validate_map)
------------------------------------------------------------------------------
[ ] Only allowed characters: '0' '1' 'N' 'S' 'E' 'W' ' '
[ ] Exactly ONE player start (N/S/E/W) - reject 0 or >1
[ ] Record player start position + facing:
    -> config.start_x, config.start_y, config.start_dir
[ ] Map must be closed / surrounded by walls:
    [ ] flood-fill from player start over '0' and player cells
    [ ] if the fill reaches the map edge OR a space -> map is open -> error
    [ ] (alt: check every walkable cell's 4 neighbours are in-bounds & not ' ')
[ ] Reject empty map (no map at all)
[ ] Make sure validation works on non-rectangular maps (hence padding above)

NOTES
------------------------------------------------------------------------------
- Identifier lines may have leading spaces and any amount of whitespace
  between the identifier and its value -> parse defensively.
- Empty lines are allowed between identifiers, but NOT inside the map body.
- The map can contain spaces that are valid (void areas), so spaces are the
  tricky part of both parsing and closure validation.
